## 2026-03-04 - [Unprotected /setup Route - Race Condition]
**Vulnerability:** The `/setup` route allowed unauthenticated creation of a superadmin. It relied solely on a client-side check (`checkIfDatabaseEmpty`) and Firestore rules allowing superadmin creation if `_setup/config` didn't exist. Because the writes (user creation, then config creation) were sequential and not batched, an attacker could continuously hit the endpoint, or hit it while a legitimate user was setting up the system, creating multiple superadmins before the `_setup/config` document was written to lock the setup down.
**Learning:** Sequential, client-driven writes to establish core security state (like initial lockouts) are vulnerable to race conditions. An attacker can exploit the window between the first write and the lock write.
**Prevention:** Always use atomic Firestore batches (`writeBatch`) when creating initial state and locks. Enforce atomicity at the `firestore.rules` level by requiring the lock document to be created in the same transaction using `existsAfter` or `getAfter` (e.g., `existsAfter(/databases/$(database)/documents/_setup/config)`).
